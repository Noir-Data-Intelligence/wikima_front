1. OBJETIVO & PRINCÍPIOS
================================================================================
Objetivo: reconstruir a interface da WiKima com um layout mais LIMPO, MODERNO e
INTUITIVO, 100% alinhado com a identidade visual da marca (pasta "WIKIMA Branding"),
com suporte a tema claro/escuro e tradução PT/EN, sobre uma base técnica moderna
e consistente.

Princípios de design:
  1. Marca primeiro — laranja WiKima como acento, navy/preto como base, ondas
     como motivo-assinatura. Nada de cores aleatórias.
  2. Consistência via tokens — zero cores hardcoded; tudo por variáveis de tema.
  3. Clareza > densidade — respiração, hierarquia tipográfica, menos ruído visual.
  4. Componível — design system de componentes reutilizáveis (shadcn/ui como base).
  5. Acessível — contraste AA, foco visível, navegação por teclado, aria.
  6. Rápido e fluido — micro-interações subtis (Framer Motion), skeletons, sem jank.
  7. Responsivo e mobile-first — a app já é PWA; manter e melhorar.

O que ESTE plano NÃO muda:
  - A camada de dados src/api/ (client REST) e os hooks TanStack Query — reaproveitados.
  - O modelo de domínio (entidades, perfis personal/professional/company).
  - A lógica de negócio existente (será re-ligada aos novos componentes).

2. ANÁLISE DO ESTADO ATUAL (baseline)
================================================================================
Dimensão do frontend:
  - ~45.780 linhas em src/ (JSX/JS).
  - 46 páginas (40 top-level + 6 admin).
  - 49 componentes shadcn/ui já instalados em src/components/ui/ (base pronta).
  - ~30+ componentes de domínio por pasta (dashboard, clients, tasks, finance…).

Dívida técnica de estilo (razão principal do redesign):
  - 408 blocos de style={{ }} inline.
  - 1.338 cores hexadecimais hardcoded (dominantes: #0F172A, #334155, #1C2D5F,
    #E97C3F, #1E293B, #B8C0D4). Palete slate/navy + laranja, próxima mas NÃO igual
    à marca oficial (#1A206A / #E88134).
  - Tokens shadcn (--background, --primary…) existem mas com valores genéricos
    (preto/branco) e são ignorados pelo código.

Tema:
  - next-themes está instalado mas praticamente não é usado (apenas 2 utilitários
    "dark:" em toda a app). NÃO existe dark/light real — a app é hardcoded escura.
  => O sistema de tema tem de ser construído de raiz.

i18n:
  - src/components/LanguageContext.jsx: dicionário plano num único ficheiro,
    ~188 chaves PT + bloco EN, função t(key), persistência em localStorage,
    deteção de idioma do browser. Consumido por 75 componentes.
  - MAS muito texto continua hardcoded diretamente nos componentes (188 chaves são
    poucas para 46 páginas). Cobertura de tradução incompleta.

Navegação / arquitetura atual (a preservar em termos de IA):
  - Layout.jsx envolve tudo em providers (Language, DemoMode, Tier, Workspace,
    UserType) — MAS "cada página renderiza a sua própria sidebar" (DashboardSidebar
    repetida por página). Não há casca (shell) partilhada. => corrigir no novo front.
  - Routing em App.jsx + pages.config.js (auto-registo de páginas).
  - Perfis de utilizador: 'personal' | 'professional' | 'company' controlam que
    módulos aparecem no menu. Existe também secção "WiKima Admin" (role === 'admin').

Bibliotecas (uso atual):
  - Já usadas: @tanstack/react-query (38), recharts (4), sonner (33), date-fns (25),
    lucide-react (ícones), react-router-dom v6.
  - Presentes mas SUButilizadas: framer-motion (0 usos), zod (0), react-hook-form (1),
    shadcn ui/table (0 usos).
  - Em falta (a adicionar): @tanstack/react-table.

Resíduos a remover:
  - src/lib/VisualEditAgent.jsx — bridge do editor visual do Base44 (postMessage
    para janela-pai). Sem função na app standalone. Remover e tirar do App.jsx.

Assets locais já presentes em public/: wikima-logo.png, auth-image-1.png,

5. ARQUITETURA DO NOVO FRONTEND
================================================================================
Estratégia: redesenho IN-PLACE no mesmo repositório, reaproveitando src/api/ e os
hooks de dados. Introduzir uma camada de design system e uma casca (shell) partilhada,
depois migrar página a página. Assim a app nunca "parte" durante a transição.

5.1 ESTRUTURA DE PASTAS (proposta)
  src/
    api/                     (mantido — REST client)
    app/
      providers.jsx          (composição de todos os providers: Query, Theme, i18n,
                              Auth, Workspace, UserType, Tier, DemoMode, Tooltip)
      router.jsx             (rotas declarativas, lazy-loading por página)
    assets/brand/            (logos SVG, ícone, ondas)
    components/
      ui/                    (shadcn base — mantido/afinado)
      brand/                 (Logo, WaveMark, WaveBackground, BrandSpinner)
      layout/                (AppShell, Sidebar, Topbar, MobileNav, PageHeader,
                              ThemeToggle, LanguageSwitcher, UserMenu, WorkspaceSwitcher)
      data/                  (DataTable [TanStack], EmptyState, StatCard, ChartCard,
                              FilterBar, Pagination, ConfirmDialog)
      forms/                 (FormField wrappers RHF, FileUpload, DatePicker, MoneyInput)
    features/                (lógica por domínio: tasks, clients, invoices, …)
    pages/                   (as páginas — reescritas, uma a uma)
    lib/                     (utils, query-client, i18n config, theme)
    styles/                  (index.css com tokens)
  Nota: a estrutura exata pode ser ajustada; o objetivo é separar UI base (ui/brand),
  casca (layout), primitivos de dados (data/forms) e domínio (features/pages).

5.2 CASCA (APP SHELL) — o maior ganho de UX
  Criar UM shell partilhado (não repetido por página):
    <AppShell>
      <Sidebar/>            (navegação por grupos e perfis; recolhível; ícone-only)
      <Topbar/>             (busca/command, WorkspaceSwitcher, ThemeToggle,
                            LanguageSwitcher, notificações, UserMenu)
      <main>
        <PageHeader/>       (título, breadcrumb, ações primárias — padronizado)
        <Outlet/>           (conteúdo da página)
      </main>
    </AppShell>
  - Sidebar deriva do menu já existente (grupos: Workspace, Clientes & Equipa,
    Trabalho, Finanças, Administração + secção WiKima Admin) e respeita os perfis
    personal/professional/company. Recolhível (expand/collapse) com ícone "W".
  - Layout público (marketing/auth) separado: <PublicLayout> com navbar + footer.

5.3 ROUTING (URLs REST — DECIDIDO)
  - Migrar para rotas com lazy() + Suspense (code-splitting por página → bundle
    atual de 2.1MB cai drasticamente).
  - Grupos de rota: público, auth, app (protegido), admin (protegido + role).
  - Guards reutilizados/limpos: ProtectedRoute (auth + onboarding), AdminRoute (role).
  - URLs modernizadas para REST/kebab-case, com parâmetros de rota em vez de query:
      /ClientProfile?id=X      -> /clients/:id
      /ProjectProfile?id=X     -> /projects/:id
      /TaskDetail?id=X         -> /tasks/:id
      /TeamMemberProfile?id=X  -> /team/:id
      /Invoices, /Tasks, ...   -> /invoices, /tasks, ... (minúsculas)
      Admin                    -> /admin/users, /admin/subscriptions, /admin/audit-logs, ...
  - Manter redirects das rotas antigas → novas para não partir links/bookmarks.
  - Auditar e atualizar todos os createPageUrl()/Link internos para as novas rotas.
