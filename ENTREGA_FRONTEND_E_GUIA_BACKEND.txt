B1. COMO O FRONTEND FALA COM O BACKEND (arquitetura src/api/)
--------------------------------------------------------------------------------
Toda a comunicação passa por uma única camada em src/api/. NÃO é preciso mexer
nos componentes/páginas — eles consomem sempre o objeto `api`:

    import { api } from '@/api/client';
    api.entities.Client.filter({ workspace_id })   // dados de negócio (CRUD)
    api.auth.me()                                    // autenticação
    api.integrations.Core.UploadFile({ file })      // integrações
    api.functions.invoke('syncGmail', payload)       // funções serverless
    api.agents.createConversation(payload)           // agentes conversacionais
    api.app.getPublicSettings()                      // metadados da app

Ficheiros da camada (todos comentados em detalhe):
    src/api/config.js       → resolve as base URLs (VITE_API_URL, default '/api').
                              Suporta apontar recursos a microserviços diferentes
                              (serviceBaseUrls) sem mudar chamadas.
    src/api/tokenStore.js   → guarda o token de auth (hoje: Bearer em localStorage).
                              ÚNICO ficheiro a mudar se optarem por cookie httpOnly.
    src/api/http.js         → wrapper fetch: get/post/put/patch/del, JSON+multipart,
                              injeta Authorization: Bearer <token>, e normaliza
                              TODOS os erros num ApiError { status, data, message }.
    src/api/resource.js     → createResource(path): CRUD REST genérico por recurso.
    src/api/entities.js     → gera um cliente REST por entidade (40 entidades).
    src/api/auth.js         → serviço de autenticação.
    src/api/integrations.js → upload de ficheiros, LLM, email/SMS, imagem, extração.
    src/api/functions.js    → invoke(name, payload) → POST /functions/{name}.
    src/api/agents.js       → conversas de agente (subscribe via polling; trocar
                              por SSE/WebSocket quando existir).
    src/api/client.js       → monta e exporta `api = { entities, auth, integrations,
                              functions, agents, app, appLogs }`.
    src/api/mock.js         → implementação MOCK usada enquanto não há backend (B10).

REGRA DE OURO PARA O BACKEND: implementem os endpoints que estas chamadas esperam
(secções B2–B5) e a app funciona sem alterar UI. Se a vossa convenção de nomes de
rotas for diferente, há UM ponto único para adaptar por tipo (ex.: `toPath` em
entities.js para o padrão dos recursos) — ver B4.


B2. CONTRATO REST — CONVENÇÃO GERAL
--------------------------------------------------------------------------------
  - Base URL: definida por VITE_API_URL (default '/api'). Em dev, o Vite faz proxy
    de '/api' para VITE_API_PROXY_TARGET (default http://localhost:8000) — logo o
    browser fala sempre same-origin e NÃO há problemas de CORS em dev.
  - Autenticação: header `Authorization: Bearer <token>` em todos os pedidos que
    tenham token guardado (ver B3/B7).
  - Corpo: JSON (Content-Type: application/json). Uploads: multipart/form-data.
  - Respostas: JSON. Content-Type deve incluir application/json.
  - Campos automáticos que o backend DEVE devolver em cada registo:
        id           (string) — chave primária
        created_date (string ISO 8601) — usado para ordenação por defeito
    (Estes nomes vêm da convenção original; manter reduz alterações no frontend.)
  - Datas: strings ISO ("2026-07-03" para date, "2026-07-03T10:00:00Z" p/ datetime).
  - Moeda/valores: números (não strings).


B3. CONTRATO REST — AUTENTICAÇÃO  (src/api/auth.js)
--------------------------------------------------------------------------------
Método frontend                         → Endpoint esperado
  api.auth.me()                          → GET  /auth/me            → devolve o User
  api.auth.updateMe(data)                → PUT  /auth/me            → User atualizado
  api.auth.loginViaEmailPassword(e, p)   → POST /auth/login         → { access_token, user }
  api.auth.register(data)                → POST /auth/register      → { access_token?, requires_otp?, user? }
  api.auth.verifyOtp(data)               → POST /auth/verify-otp    → { access_token, user }
  api.auth.resendOtp(email)              → POST /auth/resend-otp    → { ok:true }
  api.auth.resetPasswordRequest(email)   → POST /auth/reset-password/request → { ok:true }
  api.auth.resetPassword(data)           → POST /auth/reset-password → { ok:true }
  api.auth.loginWithProvider(prov, next) → REDIRECT p/ {API}/auth/oauth/{provider}?next=...
  api.auth.logout()                      → apenas limpa o token no cliente (sem chamada)

Notas:
  - O login guarda `res.access_token` automaticamente (tokenStore). O backend deve
    devolver esse campo. Para OTP, o token vem no verifyOtp.
  - `GET /auth/me` é a chamada central: o guard da app chama-a no arranque. Deve
    devolver o objeto User COMPLETO (ver campos em B6 e no schema User.jsonc),
    incluindo pelo menos: id, email, full_name, role, user_profile,
    onboarding_completed, current_workspace_id, default_workspace_id.
  - 401/403 em /auth/me = token inválido/expirado → o frontend trata como
    "auth_required" e manda para /login.
  - OAuth: o frontend faz um redirect de browser para {API}/auth/oauth/{provider}.
    O backend trata o fluxo e no fim redireciona de volta para a app (ex.: /dashboard)
    já com sessão/token. Há páginas de callback Gmail no frontend (ver B5/functions).


B4. CONTRATO REST — ENTIDADES CRUD  (src/api/resource.js + entities.js)
--------------------------------------------------------------------------------
Cada entidade é um recurso REST. O caminho é derivado do nome em kebab-case
SINGULAR:  BankAccount → "bank-account",  Client → "client",  TaskComment →
"task-comment".  (Se o backend preferir plural ou snake_case, mudar APENAS a
função `toPath` em src/api/entities.js — um só sítio.)

Para um recurso {path}, os métodos mapeiam assim:

  Método frontend                 HTTP + rota                 Resposta esperada
  ---------------------------------------------------------------------------------
  .list(sort, limit)              GET  /{path}?sort&limit     ARRAY de registos
  .filter(criteria, sort, limit)  GET  /{path}?<criteria>...  ARRAY de registos
  .get(id)                        GET  /{path}/{id}           1 registo (objeto)
  .create(data)                   POST /{path}                registo criado (c/ id)
  .update(id, data)               PUT  /{path}/{id}           registo atualizado
  .delete(id)                     DELETE /{path}/{id}         { success:true } (ou 204)

Detalhes CRÍTICOS de query (ver toQueryString em src/api/http.js):
  - `filter(criteria)` envia cada par chave=valor como query param de IGUALDADE.
      Ex.: api.entities.Task.filter({ workspace_id:'w1', status:'todo' })
           → GET /task?workspace_id=w1&status=todo
  - Arrays viram parâmetros repetidos:  { id:['a','b'] } → ?id=a&id=b
  - Objetos aninhados viram JSON string no valor do parâmetro.
  - Ordenação: parâmetro `sort`. Convenção de sinal:
        sort=-date          → descendente por 'date'
        sort=next_due_date  → ascendente
    (O frontend usa muito '-created_date', '-date', '-month', 'next_due_date'.)
  - `limit` limita o nº de resultados.
  - list()/filter() DEVEM devolver sempre um ARRAY (mesmo vazio []), NUNCA null.

Exemplos reais usados no código:
    api.entities.Expense.filter({ workspace_id }, '-date')
    api.entities.Invoice.filter({ workspace_id, status:'sent' })
    api.entities.WorkspaceMember.filter({ workspace_id, user_email })
    api.entities.Client.create({ workspace_id, name, email, ... })
    api.entities.Task.update(id, { status:'completed' })

B7. MULTI-TENANCY (workspace_id) — LER COM ATENÇÃO
--------------------------------------------------------------------------------
A app é MULTI-TENANT. Um utilizador pode pertencer a vários "workspaces"
(empresa e/ou pessoal). QUASE TODAS as entidades de negócio têm `workspace_id`.

O frontend:
  - Guarda o workspace ativo em User.current_workspace_id.
  - Envia `workspace_id` no filtro de quase todas as leituras e no corpo das criações.
    Ex.: api.entities.Task.filter({ workspace_id })
         api.entities.Client.create({ workspace_id, ... })

O backend DEVE:
  - Validar que o utilizador autenticado é membro do workspace_id pedido
    (via WorkspaceMember) e NEGAR acesso caso contrário. NÃO confiar cegamente no
    workspace_id do cliente — usar o token para autorizar.
  - Idealmente, aplicar isolamento por tenant no servidor (mesmo que o cliente
    envie workspace_id, filtrar sempre pelos workspaces a que o utilizador pertence).
  - Respeitar os papéis/permissões definidos em WorkspaceMember (owner/admin/member).

Este é o ponto de segurança mais importante da API. Tratar workspace_id vindo do
cliente como uma PREFERÊNCIA a validar, nunca como autorização.


B8. FORMATO DE ERROS ESPERADO PELO FRONTEND  (src/api/http.js + AuthContext.jsx)
--------------------------------------------------------------------------------
O http.js normaliza qualquer falha num `ApiError` com { status, data, message }.
Para respostas de erro, o backend deve devolver JSON com uma mensagem legível:
    { "message": "texto do erro" }   (ou { "error": "..." })

O frontend lê especificamente (no arranque, AuthContext):
  - err.status  → 401/403 = sessão inválida → manda para /login.
  - err.status === 403 && err.data.extra_data.reason  → casos especiais:
        reason === 'auth_required'          → precisa de login
        reason === 'user_not_registered'    → utilizador não registado nesta app
    Ou seja, para bloquear o arranque por falta de auth, devolver:
        HTTP 403  { "extra_data": { "reason": "auth_required" } }
  - status 0 = rede inacessível (o wrapper trata; útil saber para debugging).

Recomendação: usar códigos HTTP corretos (200/201, 400, 401, 403, 404, 409, 422,
500) e sempre um corpo JSON com "message".


B9. VARIÁVEIS DE AMBIENTE E COMO LIGAR O BACKEND
--------------------------------------------------------------------------------
Criar um ficheiro .env na raiz do projeto (wikima/) com:

    # URL base da API. Se começar por '/', o Vite faz proxy em dev (sem CORS).
    VITE_API_URL=/api

    # Alvo do proxy de dev (para onde '/api' é reencaminhado). Ajustar à porta
    # do vosso backend.
    VITE_API_PROXY_TARGET=http://localhost:8000

    # DESLIGAR o modo mock para usar o backend real (ver B10). Default é mock ON.
    VITE_MOCK_AUTH=false

Alternativa sem proxy (backend noutro host): pôr VITE_API_URL=https://api.wikima.com
(URL absoluto). Nesse caso o backend precisa de CORS a permitir a origem do frontend.

Microserviços: para apontar um recurso específico a outro serviço, adicionar uma
entrada em `serviceBaseUrls` em src/api/config.js (sem tocar em chamadas).

Depois de criar o .env, reiniciar `npm run dev`.


B10. MODO MOCK (como está a correr HOJE e como desligar)
--------------------------------------------------------------------------------
Hoje a app corre com um backend FALSO (src/api/mock.js), ativado por defeito:
  - Qualquer email/password faz login.
  - api.auth.me() devolve um "Utilizador Demo" (role admin, onboarding feito).
  - Todas as entidades devolvem listas vazias (o Workspace devolve 1 workspace demo).
  - Integrações/functions/agents são no-ops.

Isto existe só para navegar/validar a UI antes do backend existir. A flag é:
    MOCK_MODE = (VITE_MOCK_AUTH === 'false') ? false : true    // default: MOCK ON

PARA USAR O BACKEND REAL:
  1. Definir VITE_MOCK_AUTH=false no .env.
  2. Definir VITE_API_URL / VITE_API_PROXY_TARGET (B9).
  3. Reiniciar o dev server.
Quando o backend estiver estável, pode-se apagar src/api/mock.js e os ramos
MOCK_MODE em client.js e resource.js para remover o mock por completo.


B11. DECISÕES EM ABERTO (a equipa de backend decide)
--------------------------------------------------------------------------------
  - MECANISMO DE AUTH: hoje o frontend assume Bearer token em localStorage. Se
    preferirem cookies httpOnly (mais seguro contra XSS), mudar APENAS
    src/api/tokenStore.js (getToken devolve null) e, em http.js, acrescentar
    `credentials: 'include'` no fetch. A UI não muda.
  - CONVENÇÃO DE ROTAS: kebab-case singular por defeito (B4). Se usarem plural
    (/clients) ou snake_case (/bank_account), ajustar `toPath` em entities.js.
  - PAGINAÇÃO: o frontend usa `limit` mas ainda não paginação por offset/cursor.
    Se precisarem, é fácil estender resource.js — combinar com a equipa de frontend.
  - REALTIME (agents/mensagens): hoje é polling. Preferível SSE/WebSocket — trocar
    subscribeToConversation em src/api/agents.js quando existir endpoint.
  - PUBLIC SETTINGS: definir o que vai em /app/public-settings (feature flags,
    branding, etc.). Hoje o frontend só precisa que devolva { id, public_settings }.
